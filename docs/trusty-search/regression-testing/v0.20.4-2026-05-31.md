### Note: BM25 Circular Bias

**This snapshot is subject to BM25 circular bias (issue #123).** The test
corpus includes the query strings as literal text in the test files
(`crates/trusty-search/tests/baseline_trusty_tools.rs`). BM25 scores are
inflated for queries that appear verbatim in the codebase. This affects all
pre-#123 snapshots uniformly, making cross-version comparisons valid as long
as both snapshots use the same test suite.

## Benchmark Port Configuration

The canonical benchmark test (`baseline_trusty_tools.rs`) is hardcoded to
port 7878. During this cert, a second daemon (v0.12.3) was running on 7878.
The test file was patched in the bench worktree to target port 7879 (the
0.20.4 daemon) for this certification run. The port patch is NOT committed —
it is a measurement artefact only. The snapshot documents the measurement
accurately.

This situation arose because the machine hosts multiple daemon versions
concurrently (7878 = legacy v0.12.3 session manager instance; 7879 = current
v0.20.4 production instance). Future cert runs should use the port.lock file
to detect the live production daemon port.

### Analysis of Relevance Failure

**Query**: `usearch search query dim mismatch VectorHit`
**Expected**: file path contains `store` (i.e. `store.rs` or `hnsw_store.rs`)
**Actual top-3**:
1. `crates/open-mpm/src/tools/memory/vector_search.rs`
2. `crates/trusty-common/src/bm25_client.rs`
3. `crates/trusty-search/src/core/classifier.rs`

`store.rs` did not appear in the top-20 results for this query.

**Root cause (probable)**: Since v0.13.1, the trusty-tools monorepo has grown
substantially. The `open-mpm` crate now contains `vector_search.rs` which
implements usearch-backed vector search — a new file that matches the query
terms "usearch", "search", and "vector" strongly. This shifts the BM25 + HNSW
fusion result such that `store.rs` (the original expected file) is displaced
from the top-3.

This is a **corpus-induced regression** (new competitive content in the corpus),
not a search-quality algorithmic regression. The query was designed to anchor
on `store.rs` in v0.11.0–v0.13.1 when `open-mpm/vector_search.rs` did not exist
or was not indexed. Now `vector_search.rs` is semantically closer to the query
than `store.rs`, making it arguably the *more correct* answer.

**Regression verdict**: FLAGGED (exceeds 5 pp threshold — see Comparison table).
**Recommended action**: Update the query fragment from `store` to a more specific
anchor (e.g., `store.rs` absolute path, or a new query that uniquely targets
`store.rs`) rather than filing a bug. The underlying search quality is NOT
degraded; the corpus has grown in a way that the old fixture no longer
uniquely identifies its intended file.

### Relevance Delta

| Metric | v0.13.1 | v0.20.4 | Delta | Status |
|--------|---------|---------|-------|--------|

| Relevance queries passing | 6/6 (100%) | **5/6 (83.3%)** | **-16.7 pp** | REGRESSED |

The relevance pass rate dropped 16.7 percentage points — exceeding the 5 pp
regression threshold. See the Relevance Failure analysis above.

**Assessment**: This is a corpus-induced regression (new competitive content),
not an algorithmic regression. The "usearch search query dim mismatch VectorHit"
query was specifically designed around the v0.11.0-era corpus state. The test
fixture needs updating. The underlying search quality is appropriate — the
engine is returning the most semantically relevant file for the query given the
current corpus.

### Peak RSS Delta

| Metric | v0.13.1 | v0.20.4 | Delta | Notes |
|--------|---------|---------|-------|-------|
| Daemon peak RSS (reindex) | 2,089 MB | N/A (no reindex) | — | Not comparable |
| Daemon idle RSS (serving 66/129 indexes) | 1,543 MB | **4,044 MB** | +2,501 MB | More indexes (129 vs 66) |

RSS comparison is not meaningful for this cert because v0.13.1 measured RSS
during a fresh reindex on 66 indexes, while v0.20.4 measured idle RSS on a

### KG Growth

| Metric | v0.13.1 | v0.20.4 | Delta |
|--------|---------|---------|-------|
| KG nodes | 17,586 | 19,104 | +1,518 (+8.6%) |
| KG edges | 409,783 | 188,134 | -221,649 (-54%) |

The edge count reduction is notable: ModuleContains edges dropped from 361,986
to 149,794 (−58%). This likely reflects a KG build configuration change
between v0.13.1 and v0.20.4 (possibly a ModuleContains edge reduction or
pruning policy change, or the last indexed corpus was at a different state).
No functional impact on relevance tests was observed for the 5 passing queries.
