### Note: BM25 Circular Bias

**This snapshot is subject to BM25 circular bias (issue #123).** The test
corpus includes the query strings as literal text in the test files
(`crates/trusty-search/tests/baseline_trusty_tools.rs`). BM25 scores are
inflated for queries that appear verbatim in the codebase. This affects all
pre-#123 snapshots uniformly, making cross-version comparisons valid as long
as both snapshots use the same test suite.

### Analysis of Relevance Failure

**Query**: `usearch search query dim mismatch VectorHit`
**Expected fragment**: `vector_search`
**Actual top-1**: `crates/trusty-search/src/core/store.rs`

The fixture was updated in v0.24.1 to expect `vector_search` (after the
corpus-induced regression was first identified at v0.20.4 where the expected
fragment was `store`). In the current corpus state, `store.rs` is still
ranking above any `vector_search`-named file for this query. The test
expects `vector_search` but the top result is `store.rs`.

Root cause analysis:
- The `crates/trusty-search/src/core/store.rs` file implements the usearch
  HNSW store and contains the `VectorHit` type — making it a strong BM25
  match for the query terms "usearch", "store", "VectorHit".
- The fixture's expected fragment `vector_search` may be correct for the
  intended semantic query, but BM25 circular bias (issue #123) inflates
  `store.rs` because the query string appears literally in the test file
  which is part of the indexed corpus.
- This is an ongoing cross-version fixture issue (also flagged in v0.20.4).

**Regression verdict**: FLAGGED — same failure mode as v0.20.4 (corpus-induced
fixture mismatch), not an algorithmic regression. The test fixture for this
query needs updating to match the current corpus state.
