**Note on warm-boot stage state**: Stage states are in-memory only (not persisted to disk). After a daemon restart, all stages reset to `Pending` for normal (non-lexical-only) indexes. The search handler auto-degrades to BM25-only when `stages.semantic.status != "ready"`. This is correct by design per the Phase 1 spec — the index data on disk is intact and searches work immediately via BM25; stages advance to `Ready` only after the next reindex run.

## Phase 1 Specific Tests (#109)

### 4a. `?stage=lexical` vs auto-degraded (BM25 warm-boot)

Since all stages are `Pending` on warm boot, the daemon auto-degrades to BM25-only. Both `stage=lexical` explicit and no-stage-param produce identical results and the same `caps=[]` surface. This confirms correct graceful degradation behavior.

| Query | Auto-degraded ms | `stage=lexical` ms | Results identical | Latency delta |
|-------|-----------------|-------------------|------------------|---------------|
| `apply_archive_downrank` (exact symbol) | 6ms | 4ms | ✓ yes | -2ms |
| `symbol graph BFS expansion` (conceptual) | 3ms | 3ms | ✓ yes | 0ms |
| `struct.*Indexer` (pattern-like) | 1ms | 0ms | ✓ yes | -1ms |

**Finding**: `stage=lexical` correctly routes to BM25-only and is 1-2ms faster than the default path (avoids the HNSW capability check overhead). Results are identical because the semantic lane is not yet available. No errors or panics on the explicit `stage=lexical` param.

**Limitation**: A true hybrid vs. lexical-only delta comparison requires running after a full reindex (stages fully `Ready`). That comparison is deferred to a follow-up validation session. Per the engineer caveat, Phase 1 graceful degradation is at the search-handler level; the per-batch embed still runs lockstep with commit for non-lexical-only indexes during reindex, so the wall-clock benefit shows up primarily in `--lexical-only` indexes.

## Anomalies and Follow-ups

### Sanity Query Failure on docs-lexical-test

The CLI reported `✗ Reindex produced unhealthy index: 57 chunks (was 0), sanity query returned 0 results`. The 57 chunks are correct and search via direct HTTP worked fine. The CLI sanity check appears to be too aggressive for very small doc-only indexes (no code symbols = BM25 probe may not find content). File as a minor CLI bug.

### BM25 Circular Bias (#123) — Still Active

All v0.9.0 benchmark queries are contaminated by BM25 circular bias (queries appear as literal strings in the indexed test files). The 100% Hit@1 result is inflated. Once #123 lands with isolated benchmark queries, all baselines will be remeasured.

### get_call_chain Regression (#122) — Resolved in BM25 Mode

The v0.8.3 `get_call_chain` rank-8 regression (tracked in #122) does not reproduce in v0.9.0 BM25-only mode. The regression was specific to the HNSW candidate ordering in hybrid mode with the v0.8.3 corpus. Further investigation needed when hybrid mode is benchmarked (after full reindex).

### Embed Skipped on --lexical-only: "embedder unavailable — BM25-only mode"

The CLI output shows `Embed: SKIPPED (embedder unavailable — BM25-only mode)` for lexical-only indexes. The message is slightly misleading — the embedder is available (daemon shows `embedder: ready`); the skip is intentional because `lexical_only=true`. Minor UX issue for a future PR.
