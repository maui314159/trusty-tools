## #138 Change Summary

Commit 5541382 adds four new MCP tools (`search_lexical`, `search_semantic`, `search_kg`, `search_all`) that push intent classification to the LLM rather than the daemon. Each tool pins the daemon to a fixed lane:

| Tool | Stage pin | `expand_graph` | Prerequisite capability |
|------|-----------|----------------|-------------------------|
| `search_lexical` | `stage=lexical` | false | none (always available) |
| `search_semantic` | `stage=semantic` | false | `vector` (Stage 2) |
| `search_kg` | `stage=graph` | true | `kg` (Stage 3) |
| `search_all` | none (adaptive) | true | none (degrades gracefully) |

Legacy `search` is preserved as a back-compat alias. Total tool count: **18** (was 14 before #138).

### Tool Descriptions (verbatim from tools.rs)

**search_lexical**:
> Find code by exact symbol name, regex, or literal string. Equivalent to a fast ripgrep on the indexed codebase. Use this FIRST for any query where the user mentions a specific identifier (function name, struct name, file name) or a literal phrase. Best for: `apply_archive_downrank`, `pub fn main`, `"TODO: refactor"`, filename globs like `*.toml`. Don't use for: conceptual queries like "how does authentication work" — use `search_semantic` instead. Always available on any indexed project. Cheapest tool in this family.

**search_semantic**:
> Find code by meaning, not by literal text. Uses embedding-based similarity to retrieve chunks that semantically match the query, even when the query words don't appear in the code. Best for: "code that handles JWT verification", "the place that does community detection", "how does the embedder batch requests". Don't use for: exact symbol lookups (use `search_lexical`) or finding callers of a known function (use `search_kg`). Requires Stage 2 (embeddings) to be ready on the index — returns a STAGE_NOT_READY error with a `suggested_tools` retry hint if not. Medium cost.

**search_kg**:
> Explore code structure from a known seed — either a chunk_id (from a previous search result) or a symbol name. Returns chunks connected to the seed via `calls`, `called_by`, `contains`, `inherits` edges. Best for: "what calls `validate_token`", "what does `Authenticator` use internally", impact analysis before a refactor. Don't use for: free-text discovery (use `search_semantic`) or initial entry-point finding (use `search_lexical` first). Requires Stage 3 (symbol graph) to be ready. Returns empty if the seed is not in the index. Cheap once you have a seed.

**search_all**:
> When in doubt, use this. Runs the full hybrid pipeline (lexical + semantic + KG expansion) and merges results via RRF. More expensive than the targeted tools but catches edge cases. Use when: your query has both literal symbols AND conceptual phrasing ("find the `AuthValidator` that handles refresh tokens"), or when you've tried the targeted tools and they didn't surface what you need. Always available; gracefully degrades to whatever lanes are ready. When called without `index_id`, falls back to legacy cross-project fan-out behaviour (issue #10) — provide `index_id` for the per-index hybrid path.

## Step 2 — STAGE_NOT_READY Happy Path

Created a lexical-only probe index (`stage-not-ready-probe`) and called `search_semantic` against it via MCP HTTP transport.

### Probe Index Status (after creation, before reindex)

```json
{
  "lexical_only": true,
  "search_capabilities": [],
  "stages": {
    "graph":    {"status": "skipped"},
    "lexical":  {"status": "pending"},
    "semantic": {"status": "skipped"}
  }
}
```

### STAGE_NOT_READY — Bare Method Form

```json
{
  "jsonrpc": "2.0",
  "id": 3,
  "error": {
    "code": -32010,
    "message": "search_semantic requires Stage 2 (embeddings), which is not yet ready on index 'stage-not-ready-probe'. Current stages: lexical=Pending, semantic=Skipped, graph=Skipped. Suggested: use search_lexical, search_all, or wait for the reindex to complete.",
    "data": {
      "current_stages": {
        "graph":    {"status": "skipped"},
        "lexical":  {"status": "pending"},
        "semantic": {"status": "skipped"}
      },
      "error_code": "STAGE_NOT_READY",
      "suggested_tools": ["search_lexical", "search_all"]
    }
  }
}
```

### STAGE_NOT_READY — `tools/call` Form

```json
{
  "jsonrpc": "2.0",
  "id": 4,
  "result": {
    "isError": true,
    "content": [{"type": "text", "text": "search_semantic requires Stage 2 (embeddings), which is not yet ready on index 'stage-not-ready-probe'. Current stages: lexical=Pending, semantic=Skipped, graph=Skipped. Suggested: use search_lexical, search_all, or wait for the reindex to complete."}],
    "_meta": {
      "error_code": "STAGE_NOT_READY",
      "current_stages": {
        "graph":    {"status": "skipped"},
        "lexical":  {"status": "pending"},
        "semantic": {"status": "skipped"}
      },
      "suggested_tools": ["search_lexical", "search_all"]
    }
  }
}
```

**Confirmed**:
- Bare-method form: `error.code = -32010` (STAGE_NOT_READY_CODE), `error.data.error_code = "STAGE_NOT_READY"`, `suggested_tools = ["search_lexical", "search_all"]`
- `tools/call` form: `isError = true`, `_meta.error_code = "STAGE_NOT_READY"`, `_meta.current_stages.semantic.status = "skipped"`
- Both forms agree on `suggested_tools`: `["search_lexical", "search_all"]` (correct: no vector capability, so `search_semantic` not suggested)
- Probe index deleted after test

## Step 3 — Per-Tool Benchmark (Synthetic Corpus)

The `benchmark_synthetic.rs` harness exercises all three stage lanes via the HTTP search endpoint (equivalent to calling `search_lexical` / `search_semantic` / `search_kg` via MCP, since each tool pins the same `stage` + `expand_graph` combination). The harness uses the `synthetic-benchmark` index with the glyphwarpen-observatory corpus (47 files, 298 chunks).

Corpus reindex time: **3.5s**. All three stages reached `ready` before queries began.

Raw output saved to `/tmp/ts-bench-v0100.json`.

The #138 change added tool dispatching and pre-flight status checks but did not modify any retrieval logic — so identical numbers are the expected and correct outcome.

### Note on search_semantic vs v0.9.1 "hybrid"

In v0.9.1 the "hybrid" mode used adaptive routing that would include KG expansion when the daemon's intent classifier decided to use it. In v0.10.0, `search_semantic` pins `stage=semantic, expand_graph=false` — it explicitly excludes KG expansion. On this small corpus (298 chunks, limited KG signal), the practical difference is zero: the KG lane contributed no measurable lift in v0.9.1 "hybrid" either. The theoretical risk (that some query previously hitting via KG expansion in v0.9.1 would miss via `search_semantic` in v0.10.0) is confirmed not to apply to any of the 19 test queries.

This validates the #138 design decision: `search_semantic` is a pure BM25+HNSW lane, and `search_kg` / `search_all` are the correct tools when KG expansion is desired. The A/B comparison between `search_semantic` (no KG) and `search_all` (adaptive KG) is now cleanly expressible as a side-by-side tool comparison, which was the stated goal of #128.

## Caveats

1. **BM25 circular bias OK on synthetic corpus**: confirmed by design. The glyphwarpen-observatory corpus contains no query string literals — BM25 lift is from token overlap with the actual code.
2. **`search_all` has no dedicated benchmark column** in the v0.2.0 harness. It is equivalent to hybrid mode (no stage pin). A future harness v0.3.0 could add a column.
3. **Stage 3 KG signal is unmeasurable on 298-chunk corpus**: `search_kg` and `search_semantic` produce identical Hit@K. On a larger corpus with rich cross-file call chains (e.g. trusty-tools itself), KG expansion provides material lift on `Usage` queries.
4. **Single run**: p50 latency estimates; tail latency not characterized.
