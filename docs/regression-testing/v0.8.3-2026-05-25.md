## Anomalies & Investigation

### `get_call_chain` Regression (#122)

One query (`fn get_call_chain`) behaved unexpectedly in v0.8.3:
- **v0.8.1**: Rank 4 (within top-5)
- **v0.8.3**: Rank 8 (outside top-5)

**Cause (under investigation)**: Doc indexing (#118) added 3,800 chunks containing keyword-heavy prose. BM25 re-weighting now favors doc matches over source code for certain queries. Tracked in #122.

**Workaround**: Code-mode filtering (per #118) can suppress doc matches if needed; user-facing ranking needs tuning.

### HNSW Candidate Cap Optimization (#117)

Reduced vector-retrieval candidate cap from 8→4 (adaptive reranking still fires at 8):
- **Expected impact**: Faster vector retrieval, minimal quality loss
- **Observed**: Sub-linear latency growth (p50 +2 ms despite +23% corpus)
- **Hit@5 improvement**: Likely due to better re-ranking cutoff; worth deeper analysis

## Caveat: BM25 Circular Bias (#123)

⚠️ **All results in this snapshot are contaminated by BM25 circular bias**:

The trusty-tools repository contains the 14-query suite in `crates/trusty-search/tests/baseline_trusty_tools.rs` as literal strings. When trusty-tools is indexed, BM25 matches the queries directly, artificially inflating Hit@1 and Hit@5.

**Impact on v0.8.3**: Now that docs are indexed (PR #118), the benchmark queries appear as:
- Literal strings in test files (source code)
- Implicit copies in documentation prose (if docs reference benchmark concepts)

This makes the bias **harder to measure** but potentially **more severe**. Once #123 lands, all baselines will be re-measured with the queries isolated.

**Trend validity**: Despite the bias, trends between v0.8.1 and v0.8.3 remain valid because the contamination mechanism is consistent. The Hit@5 improvement (+7.1 pp) is real; the Hit@1 regression (-1.9 pp) is real; the absolute percentages should not be trusted until #123 is fixed.
