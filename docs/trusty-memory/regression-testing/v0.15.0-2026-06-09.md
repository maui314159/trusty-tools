## Concurrent Performance Suite: Attempt and Failure

The canonical concurrent-performance suite for trusty-memory lives at
`crates/trusty-memory/tests/concurrent_perf.rs`. It is gated behind
`#[ignore]` and requires a live daemon.

### Command Attempted

```bash
cargo test -p trusty-memory --test concurrent_perf -- --include-ignored --nocapture
```

### Result

```
test result: FAILED. 0 passed; 4 failed; 0 ignored; 0 measured; 0 filtered out; finished in 0.34s
```

All four `#[ignore]`-tagged tests (`test_http_concurrent_reads`,
`test_http_concurrent_rw`, `test_http_burst`, `test_http_sustained_load`)
failed with:

```
palace_create failed: Object {
  "error": Object {
    "code": Number(-32603),
    "message": String("no project root found at or above '/'. Use 'personal' for
                       memories not tied to a project, or run from inside a project
                       directory that contains a .git file, Cargo.toml, pyproject.toml,
                       or package.json.")
  },
  ...
}
```

### Root Cause

The `concurrent_perf` test suite provisions UUID-named test palaces via
`palace_create`. In v0.15.0, `palace_create` resolves the project root from
the **daemon's** working directory, not the test client's. The production
daemon is managed by `launchd` (macOS LaunchAgent) and is launched with
`WorkingDirectory` set to `/` — a path with no project markers (no `.git`,
no `Cargo.toml`).

The `concurrent_perf.rs` suite was designed to run against a daemon started
from within a project directory. When `palace_create` is called without a
valid project context, v0.15.0 rejects the request with the error shown
above.

**This is not a daemon bug.** The `personal` palace (which requires no
project root) works correctly for non-project-scoped operations. The test
fixture needs to use `personal` as the palace name, or the tests need to
be run against a daemon started from a project directory.

**Assessment**: The concurrent_perf suite cannot run against the
production launchd-managed daemon as of v0.15.0. A follow-up ticket is
recommended to either:
1. Add a `--palace personal` override to `concurrent_perf.rs`, or
2. Document the required daemon startup configuration for the suite.

## Direct HTTP Performance Baseline

Because the canonical suite cannot run, direct HTTP measurements were taken
using Python `urllib` + threading against the live daemon on port 7070.
These measurements use the `personal` palace (no project-root required) and
represent the v0.15.0 production performance envelope.

### Command

```python
# Python 3.x urllib + threading direct HTTP benchmark
# Queries against http://127.0.0.1:7070/rpc
# POST JSON-RPC 2.0 requests to memory_recall and palace_list endpoints
```

### Results

| Method | N | min | mean | p50 | p95 | p99 | max |
|--------|---|-----|------|-----|-----|-----|-----|
| `memory_recall` (sequential) | 20 | 0.3 ms | 0.6 ms | 0.3 ms | 5.2 ms | 5.2 ms | 5.2 ms |
| `palace_list` (sequential) | 20 | 1.4 ms | 7.4 ms | 3.4 ms | 95.7 ms | 95.7 ms | 95.7 ms |
| `memory_recall` (20-concurrent, threads) | 20 | — | — | 43.8 ms | 77.8 ms | 77.8 ms | 77.8 ms |

**Wall clock for 20-concurrent recall burst**: 80 ms
**Concurrent throughput**: 250.1 req/s
**Errors**: 0 / 20

### Key Observations

1. **`memory_recall` sequential latency is exceptional** — sub-millisecond
   p50 (0.3ms warm), indicating the BM25 + HNSW retrieval pipeline is
   running hot with minimal overhead.

2. **`palace_list` has higher variance** — p95 of 95.7ms vs p50 of 3.4ms.
   This is likely due to the palace list scan requiring a full redb read
   across all registered palaces.

3. **20-concurrent reads complete in 80ms** — 250 req/s throughput with
   zero errors. The daemon handles concurrent reads without contention
   at this scale.

4. **RSS growth under load**: ~4MB delta (199 MB → 203 MB) across all
   benchmark activity, indicating no memory leak at this scale.

## Caveats

- **Sequential recall p50 of 0.3ms** should be interpreted with caution: the
  measurement is after a warm-up request and reflects a warm BM25 index with
  a small personal palace. Latency will be higher for large palaces or cold
  starts.
- **`palace_list` p95 of 95.7ms** may reflect a single outlier in 20 samples;
  a larger N sample would produce a more stable estimate.
- The `concurrent_perf.rs` suite (the canonical benchmark) could not run
  against the production daemon — see "Concurrent Performance Suite" section
  above for full root cause analysis.
