### Performance Analysis

**Fast path (< 1 ms)**:
- `analyzer_health` (0.42ms p50) and `list_facts` (0.14ms p50) are
  effectively in-process lookups — health returns a struct, facts queries
  the empty redb FactStore.

**Analysis pass (600–750 ms)**:
- `complexity_hotspots` (667ms) and `find_smells` (701ms) require fetching
  all 28,119 chunks from trusty-search via `GET /indexes/trusty-tools/chunks`,
  running the tree-sitter AST complexity/smell analysis, and ranking results.
  The ~700ms p50 reflects the chunk fetch network + tree-sitter parse time.

**Heavy analysis pass (1,100–1,600 ms)**:
- `analyze_quality` (1,118ms p50) runs the full analysis pipeline and
  aggregates across all files. The higher p99 (1,615ms) reflects GC/memory
  pressure variance during large corpus scans.

**Graph and clustering (2,300–5,300 ms)**:
- `suggest_refactors` (2,330ms) runs complexity analysis + scoring.
- `list_entities` (3,009ms p50, 5,319ms p99) is the most expensive tool —
  it extracts all CALLS/Contains graph entities from the 28,119-chunk corpus.
  The high p99 (5.3s) suggests occasional GC pauses or memory pressure on
  the 59MB+ entity graph.

### Zero Error Rate

All 350 tool calls (7 tools × 50 iterations) completed successfully with
zero errors. This confirms the daemon is stable under repeated sequential
load against the full trusty-tools corpus.

## Caveats

1. **Corpus size**: All latency measurements reflect the full trusty-tools
   corpus (28,119 chunks). Smaller corpora will be faster; larger corpora
   will be slower, approximately linearly for chunk-fetch-dominated tools.

2. **`list_entities` p99 variance**: The 5,319ms p99 vs 3,009ms p50 gap
   indicates memory pressure or GC pauses during entity extraction from the
   59MB+ entity graph. This may be an optimization target for v0.5.x.

3. **`pnpm` warning during build**: The `pnpm install failed` warning during
   `cargo test` compilation is cosmetic — the test binary does not need the
   UI. Set `SKIP_UI_BUILD=1` to suppress it.

4. **Test binary vs production daemon**: The `full_stdio_suite` spawns a new
   child process on port 17879, separate from the production daemon on 7879.
   Both target the same trusty-search index. Performance numbers reflect the
   test child, not the production daemon (the production daemon may be warmer
   due to persistent connections).

5. **Apple Silicon CoreML**: trusty-search (which trusty-analyze calls for
   corpus fetch + embeddings) uses CoreML (ANE) acceleration on this host.
   Results may differ on non-Apple-Silicon hosts or without the CoreML
   execution provider.
